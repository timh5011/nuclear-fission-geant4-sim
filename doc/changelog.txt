⏺ Two separate things bundled into that warning — let me unpack them.                                                                                                                                   
                                                                                                                                                                                                      
  1. What "critical mass" means here
                                                                                                                                                                                                        
  A bare sphere of pure ²³⁵U at 19.1 g/cm³ goes critical (k_eff = 1, self-sustaining chain reaction) at about 47–52 kg, which is a sphere of radius ~8.6 cm or a cube ~17 cm on a side. Above that,     
  every fission produces on average more than one fission in the next generation → exponential growth. Below it, neutrons leak out faster than they can cause new fissions. (For reference: Little Boy  
  used ~64 kg of HEU.)                                                                                                                                                                                  
                  
  Your current world is a 20 cm cube (10 cm half-side), so you couldn't even fit a critical mass in the world volume — but you could get uncomfortably close to it (e.g. a 5 cm cube of pure ²³⁵U is    
  ~2.4 kg, well below critical, but a 10 cm cube is ~19 kg, getting noticeable neutron multiplication even though still subcritical).
                                                                                                                                                                                                        
  2. Why this matters for Geant4 specifically                                                                                                                                                           
   
  Geant4's HP package is a radiation transport code, not a criticality code. Specialized criticality codes (MCNP-K, KENO, Serpent) use k-eigenvalue iteration and fission-source convergence algorithms 
  that Geant4 does not implement. Geant4 just does analog Monte Carlo: it tracks every secondary neutron from every fission until it escapes, gets captured, or falls below threshold.
                                                                                                                                                                                                        
  Two practical consequences if you make the target large enough that secondaries cause more fissions:                                                                                                  
   
  - Performance. Each prompt fission neutron that fissions again produces ~2.4 more neutrons. If your geometry has k_eff close to 1, a single primary can trigger thousands of secondary fissions within
   one event — runtime per event blows up, memory blows up, the OGL viewer chokes on track storage. With k_eff > 1, runtime per event is unbounded.
  - Validity. Geant4 wasn't validated for this regime. Subtle issues (delayed-neutron treatment, fission spectrum sampling, multi-generation tally bias) that don't matter for sub-mfp foils start      
  mattering. Results may look reasonable and be wrong.                                                                                                                                                  
   
  3. Where the safe zone is                                                                                                                                                                             
                  
  - Sub-cm cubes (e.g. 5 mm cube, 5 × 5 × 1 mm slab): mass ~tens to hundreds of mg, k_eff effectively 0, every primary causes at most ~1 fission, secondaries leak. This is the regime where Geant4 is  
  doing what it's good at.
  - ~1 cm cube (~19 g): still very subcritical, but you'll start seeing the occasional secondary fission inside the bulk. Still fine, just not single-interaction-clean anymore.                        
  - Several cm cubes: starting to enter "you're modeling a small fission assembly" territory — runtime grows, results need to be interpreted carefully, and you're past what this code was set up to do.
                                                                                                                                                                                                        
  So the practical recommendation: if you go to a cube, keep it ≤ a few mm on a side. That's plenty of mfp for ~100% interaction probability per primary, but small enough that secondary neutrons leak 
  instead of multiplying, which keeps each event clean and Geant4 in its validated regime.

  UIUC — inherits from FTFP_BERT                                                                                                                                                                        
                                                                                                                                                                                                        
  class MyPhysicsList : public FTFP_BERT { ... }
                                                                                                                                                                                                        
  FTFP_BERT is a Geant4 reference physics list — a pre-assembled bundle that already includes EM, hadronic, decay, and ion physics internally. By inheriting from it, the UIUC sim gets all of that for 
  free before the constructor even runs. The three RegisterPhysics calls in Physics.cc then add on top of that inherited baseline (EM standard, optical, decay — most of which FTFP_BERT already        
  provides, so there's some redundancy there).                                                                                                                                                          
                                                                                                                                                                                                        
  The split between .hh (declaration only: MyPhysicsList();) and .cc (definition) is the conventional C++ pattern — the class is declared in the header, the constructor body lives in the .cc.         
                                                                                                                                                                                                        
  ---                                                                                                                                                                                                   
  Nuclear fission sim — inherits from G4VModularPhysicsList                                                                                                                                           
                                                                                                                                                                                                        
  class MyPhysicsList : public G4VModularPhysicsList { ... }
                                                                                                                                                                                                        
  G4VModularPhysicsList is the bare base class — it does nothing on its own. Every process must be registered explicitly. The constructor in Physics.hh hand-picks exactly the modules needed: HP       
  neutron transport (QGSP_BIC_HP), hadronic elastic HP, radioactive decay, ion physics, etc. This gives you precise control, which matters here because you specifically need the High Precision (HP)
  neutron data library for the U-235 fission cross-sections below 20 MeV — FTFP_BERT uses BERT cascade which doesn't give you that.                                                                     
                  
  The .cc being just #include "Physics.hh" is a consequence of the inline constructor: since the constructor body is written inside the class definition in the header, there's nothing left to define  
  in the .cc.
                                                                                                                                                                                                        
  ---             
  Key practical difference: FTFP_BERT is a convenient turnkey list suited to the UIUC muography use case (muons through rock/concrete — no need for thermal neutron cross-sections). The fission sim
  needs the HP neutron stack, so it builds from scratch with G4VModularPhysicsList and explicitly registers QGSP_BIC_HP + HadronElasticPhysicsHP. You can't easily get that out of FTFP_BERT.

  The fission sim is clean. The UIUC one has a real issue worth knowing about.                                                                                                                          
                                                                                                                                                                                                        
  FTFP_BERT already internally registers EM physics and decay physics. Then the UIUC constructor does this:                                                                                             
                                                                                                                                                                                                        
  RegisterPhysics(new G4EmStandardPhysics());   // FTFP_BERT already has this                                                                                                                           
  RegisterPhysics(new G4OpticalPhysics());       // fine — FTFP_BERT doesn't include this                                                                                                               
  RegisterPhysics(new G4DecayPhysics());         // FTFP_BERT already has this                                                                                                                          
                                                                                                                                                                                                        
  When you register a physics module whose name collides with one already registered, Geant4 prints a warning and deletes the old one, replacing it. So the UIUC sim is silently downgrading its own EM 
  physics: FTFP_BERT uses G4EmStandardPhysics_option4 internally (higher accuracy), and that explicit G4EmStandardPhysics() call replaces it with the baseline option0. For muography through bulk
  material it probably doesn't matter much in practice, but it's unintentional.                                                                                                                         
                  
  The only registration in that constructor that's actually doing useful work is G4OpticalPhysics() — that's the one thing FTFP_BERT doesn't bring. The other two calls are redundant at best, a silent 
  downgrade at worst.
                                                                                                                                                                                                        
  The fission sim avoids this entirely by starting from G4VModularPhysicsList and being explicit about everything it registers — no hidden inherited state to conflict with.
